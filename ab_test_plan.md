# План A/B-тестирования моделей кредитного скоринга

## 1. Концепция A/B-теста

### Цель теста
Сравнить две версии модели прогнозирования дефолта по кредитным картам:
- **Контрольная группа (A)**: текущая модель v1 (LogisticRegression с пайплайном предобработки).
- **Тестовая группа (B)**: новая модель v2 (GradientBoostingClassifier с тем же пайплайном).

Цель — определить, обеспечивает ли модель v2 статистически значимое улучшение качества прогнозирования дефолта по сравнению с v1, и принять решение о переводе всего трафика на новую версию.

### Способ разделения трафика

Используется **случайное равномерное распределение 50/50** на уровне запросов (request-level randomization):

1. Каждый входящий запрос к эндпоинту `/predict` получает случайный идентификатор (UUID).
2. На основе хэша UUID запрос с вероятностью 50% направляется на модель v1, с вероятностью 50% — на модель v2.
3. Распределение проверяется ежедневно: доля запросов к каждой модели должна находиться в диапазоне 48–52%. При отклонении тест приостанавливается для диагностики.

**Почему 50/50?**
- Максимальная статистическая мощность при заданном общем объёме трафика.
- Равное время накопления данных по обеим группам.
- Минимизация риска для бизнеса: даже если v2 окажется хуже, пострадает только половина заявок.

**Альтернативы (для будущих тестов):**
- 90/10 (канареечное развёртывание) — для первых тестов рискованных изменений.
- 80/20 — когда новая модель уже показала хорошие результаты на исторических данных.

### Продолжительность теста

**Минимальная продолжительность: 14 дней (2 недели).**

Обоснование:
1. **Недельная сезонность.** Заявки на кредиты имеют выраженный недельный цикл (больше заявок в начале месяца и в будни, меньше в выходные и конце месяца). Две недели покрывают два полных цикла.
2. **Накопление выборки.** При среднем потоке ~500 заявок в день за 14 дней накопится ~7000 заявок (по ~3500 на группу), что достаточно для достижения статистической мощности 80% при обнаружении эффекта размера Cohen's h = 0.1 (малый эффект).
3. **Максимальная продолжительность: 30 дней.** Если за 30 дней значимого эффекта не обнаружено, тест останавливается с выводом "различий не выявлено".

**Досрочная остановка возможна при:**
- Явном ухудшении метрик v2 на >10% относительно v1 (срабатывает стоп-правило).
- Технических сбоях в одной из групп (ошибка логирования, падение сервиса).

## 2. Метрики и анализ

### 2.1. Основная метрика: F1-score для класса дефолта

**Определение:** Гармоническое среднее Precision и Recall для класса 1 (дефолт).

**Формула:** F1 = 2 * (Precision * Recall) / (Precision + Recall)


**Почему F1-score?**
- В задаче кредитного скоринга классы несбалансированы (~22% дефолтов). Accuracy вводит в заблуждение.
- F1-score одновременно учитывает и Precision (качество положительных прогнозов), и Recall (полноту выявления дефолтов).
- Это стандартная метрика для несбалансированной бинарной классификации в финансовой индустрии.

**Как рассчитать:**
Для каждой группы (A и B) независимо:
1. Собрать все предсказания модели и фактические исходы (дефолт/не дефолт) за период теста.
2. Построить матрицу ошибок (confusion matrix).
3. Вычислить Precision и Recall для класса 1.
4. Рассчитать F1-score.

### 2.2. Дополнительная метрика: Recall (полнота)

**Определение:** Доля реально дефолтных клиентов, которых модель правильно идентифицировала.

**Формула:** Recall = TP / (TP + FN)

где TP — true positive (правильно предсказанный дефолт), FN — false negative (пропущенный дефолт).

**Обоснование выбора Recall вместо Precision:**

В кредитном скоринге **пропуск дефолта (FN) обходится банку значительно дороже**, чем ложный отказ хорошему клиенту (FP):
- **FN (пропущенный дефолт):** Банк выдаёт кредит клиенту, который не вернёт деньги. Прямые убытки = сумма кредита × LGD (Loss Given Default, обычно 40–60%). Для кредита в 100 000 руб. убыток составит 40 000–60 000 руб.
- **FP (ложный отказ):** Банк отказывает хорошему клиенту. Убыток = упущенная прибыль от процентов (обычно 10–20% от суммы кредита в год). Для кредита в 100 000 руб. упущенная прибыль ≈ 10 000–20 000 руб.

Таким образом, **стоимость FN в 2–6 раз выше стоимости FP**. Максимизация Recall напрямую снижает финансовые потери банка.

**Дополнительно отслеживаются:**
- **Precision** — чтобы убедиться, что рост Recall не происходит за счёт катастрофического падения качества положительных прогнозов.
- **Approval Rate** — доля одобренных заявок. Резкое падение Approval Rate при росте Recall может сигнализировать о слишком агрессивной модели.
- **Expected Loss** — бизнес-метрика: `EL = PD × LGD × EAD`.

### 2.3. Метод статистического анализа

#### 2.3.1. Выбор статистического теста

**Для Recall и Precision (доли/пропорции): двухвыборочный z-test для пропорций.**

Обоснование:
- Recall и Precision — это доли (число от 0 до 1).
- При достаточно большой выборке (n > 30 в каждой группе, что гарантировано при 3500+ заявок) распределение выборочной доли приближается к нормальному (центральная предельная теорема).
- Z-test для пропорций — стандартный и наиболее мощный тест для сравнения двух долей.

**Формула z-статистики:** z = (p_B - p_A) / sqrt(p_pool * (1 - p_pool) * (1/n_A + 1/n_B))

где:
- p_A, p_B — значения метрики в группах A и B.
- p_pool — объединённая доля: (TP_A + TP_B) / (n_A + n_B).
- n_A, n_B — размеры выборок.

**Для F1-score (составная метрика): bootstrap-метод.**

Обоснование:
- F1-score не имеет простого параметрического распределения (это функция от двух долей).
- Bootstrap не требует предположений о форме распределения.
- Алгоритм:
  - Из выборки группы A случайно выбираем n_A наблюдений с возвращением (bootstrap-выборка).
  - Аналогично для группы B.
  - Считаем F1_A и F1_B, вычисляем разницу Δ = F1_B - F1_A.
  - Повторяем шаги 1–3 N = 10000 раз.
  - Строим эмпирическое распределение Δ.

#### 2.3.2. Построение доверительных интервалов

**Для z-test (Recall/Precision):** 95% CI = (p_B - p_A) ± 1.96 * SE

где SE — стандартная ошибка разницы долей.

**Для bootstrap (F1-score):**
- 95% доверительный интервал = [2.5-й перцентиль распределения Δ, 97.5-й перцентиль].
- Метод: percentile bootstrap (наиболее устойчивый к асимметрии распределения).

**Интерпретация:**
- Если 95% CI не содержит 0 → разница статистически значима на уровне α = 0.05.
- Если 95% CI содержит 0 → разница не значима, нельзя утверждать, что одна модель лучше другой.

#### 2.3.3. Критерий успешности теста

Тест считается **успешным** (модель v2 внедряется вместо v1) при одновременном выполнении **всех** условий:

1. **Статистическая значимость:** p-value < 0.05 для основной метрики (F1-score).
2. **Направление эффекта:** F1-score(v2) > F1-score(v1) (разница положительная).
3. **Практическая значимость:** относительное улучшение F1-score ≥ 3% (порог установлен исходя из стоимости внедрения новой модели).

relative_lift = (F1_B - F1_A) / F1_A >= 0.03

4. **Отсутствие деградации по дополнительным метрикам:**
- Recall(v2) >= Recall(v1) (не ухудшился).
- Approval Rate(v2) >= Approval Rate(v1) × 0.95 (не упал более чем на 5%).

**Сценарии результатов:**

| Сценарий        | Условие                         | Решение                                                  |
|-----------------|---------------------------------|----------------------------------------------------------|
| Полный успех    | Все 4 условия выполнены         | Внедрить v2, остановить v1                               |
| Частичный успех | F1 значимо выше, но Recall упал | Доработать v2 (настроить threshold)                      |
| Нет различий    | p-value >= 0.05                 | Оставить v1, собрать больше данных или изменить гипотезу |
| Провал          | F1 значимо ниже                 | Отклонить v2, исследовать причины                        |

## 3. План реализации в коде

### Архитектура A/B-теста

1. **Роутер запросов** (`ab_router.py`): принимает запрос, генерирует UUID, по хэшу направляет на v1 или v2.
2. **Две версии модели**: `models/pipeline_v1.pkl` и `models/pipeline_v2.pkl`.
3. **Логирование результатов**: каждый запрос логируется с полями:
- `experiment_id`: идентификатор теста.
- `group`: 'A' или 'B'.
- `prediction`: прогноз модели.
- `probability`: вероятность дефолта.
- `actual`: фактический исход (заполняется постфактум через 30 дней).
4. **Аналитический скрипт** (`ab_test_analysis.py`): загружает логи, считает метрики, проводит статистические тесты, выводит отчёт.

### Минимальный объём выборки (power analysis)

Для обнаружения эффекта размера h = 0.1 (малый эффект по Cohen) при:
- α = 0.05 (двусторонний тест),
- мощность = 0.80,
- соотношение групп 1:1,

требуется **n ≈ 1570 наблюдений на группу** (рассчитано по формуле для z-test пропорций).

При потоке 500 заявок/день и распределении 50/50 это достигается за **~6 дней**. Мы берём 14 дней для учёта сезонности.
